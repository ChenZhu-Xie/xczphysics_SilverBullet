
# 突然发现，Excel 是一个绝佳的 [[🤔 Daydream/⭐ P 渐进式 Web 应用 A]]，难道不是么？

每个单元格，要计算其 value 时，会自动去 grep 其“依赖项” 对应的 单元格们，如果这些 “依赖项” 们还有 “依赖项”，则继续找它们的下家，直到遇到末端，并先计算它们的值 value，然后再将这些 values 代入 import 它们的“包”们中，并计算它们，以及后续依赖它们的单元格们。

## Excel(≈ SilverBullet) vs Python

这种 “_自动找祖先” + “树状链式传递_” 式 的 app，即“**拥有单元格对象，并单元格对象自动分析依赖关系 + 广播式更新值**”的 excel，就其 _原子积木_ 层面 而言，非常像 SilverBullet 的渐进生成的理念，甚至==有时候在编码+可视化数据方面，便捷性 比 python 还强一点==，比如 “4.风向参照正对对方的自身坐标系.xlsm” 中的 **一个 data 变了，就链式 “实时更新 6 张图像”** 。这在 python 中可以做到，但“没法那么简单地做到”。

# auto-matic info grepper and distributor

昨天晚上睡前我意识到了：愚钝又敏锐的我（愚钝是因为现在才识别出来，敏锐是因为我竟然识别出来了）似乎又识别出了一些重复的、可自动化的 pattern 和 流程 process.

但 自动化的成本、投入产出比 还需要 进一步 评估，以决定是否最终实施计划 进行自动化。

昨天我不遵循 routine 地，Inspect 了一下 QQ 群 “笔记软件折腾群”，发现里面的质量出奇地高（因为 全是 人类程序猿），还特地记录下来了他们用的一些工具（名及其链接）到 Tana。

这个偶然事件，让我意识到，我应该已经错过了这个群里的许多有用的消息。这对信息敏感的我应该是很难接受的。

考虑到该群未来也（很大概率会）连续产生高质量的、我希望第一时间接收的信息量（尽管已经不再是一手信息，因为是通过人类转述的，而这些转述的高质量人类才是获得一手信息的），而我一个人又没法实时地监督该群的每一条消息，并过滤其中水群的无用消息。

这意味着，有一些额外的自动化定时/事件触发的空白 routine 需要填补。这就是结论：auto-matic infomation grepper + filter。
